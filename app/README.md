# Unscramble Game

## Краткое описание игры

**Unscramble Game** — это Android-приложение-игра, разработанное в рамках лабораторной работы №13-14. Цель игры — угадать исходное слово из перемешанных букв.

Приложение демонстрирует использование архитектурных компонентов Android Jetpack, в частности **ViewModel** и **StateFlow**, для управления состоянием пользовательского интерфейса (UI) и сохранения данных при изменении конфигурации устройства (поворот экрана, смена темы).

## Технологии

Проект разработан с использованием следующих технологий и библиотек:

*   **Kotlin** — основной язык программирования.
*   **Jetpack Compose** — современный набор инструментов для построения пользовательского интерфейса.
*   **ViewModel** — компонент архитектуры Android Jetpack для хранения и управления данными, связанными с UI, с учётом жизненного цикла.
*   **StateFlow** — reactive stream из Kotlin Coroutines, используемый для наблюдения за состоянием (State) в ViewModel.
*   **JUnit** — фреймворк для написания и запуска модульных тестов.

## Функциональность приложения

*   **Генерация слов**: Случайный выбор слова из предустановленного списка и перемешивание его букв.
*   **Проверка ответа**: Сравнение введённого пользователем слова с исходным словом без учёта регистра.
*   **Подсчёт очков**: Начисление фиксированного количества очков (например, 20) за каждый правильный ответ.
*   **Счётчик вопросов**: Отображение текущего номера вопроса из общего количества (например, "Вопрос 1 из 10").
*   **Обработка ошибок**: Отображение сообщения об ошибке (например, красная надпись) при неправильном вводе.
*   **Пропуск слова**: Возможность пропустить текущее слово без изменения счёта.
*   **Завершение игры**: Отображение диалогового окна с итоговым счётом после прохождения всех слов.
*   **Сохранение состояния**: Использование **ViewModel** гарантирует, что все данные игры (счёт, текущее слово, прогресс) сохраняются при повороте экрана.

## Контрольные вопросы

### 1. Что такое ViewModel и зачем он нужен?

**ViewModel** — это класс из Android Jetpack, предназначенный для хранения и управления данными, связанными с пользовательским интерфейсом.

*   **Какую проблему решает ViewModel?**
    *   Основная проблема, которую решает ViewModel — это сохранение данных при изменении конфигурации устройства, например, при повороте экрана. В обычных условиях Activity пересоздаётся, и все данные, хранящиеся в ней, теряются. ViewModel живёт дольше, чем Activity, и автоматически связывается с жизненным циклом, что позволяет данным "пережить" пересоздание интерфейса.
*   **Что происходит с данными при повороте экрана без ViewModel?**
    *   Без ViewModel, при повороте экрана, Android уничтожает текущую Activity и создаёт новую. Все переменные экрана (например, счёт, введённый текст) теряются, и приложение возвращается в исходное состояние.

### 2. В чём разница между MutableStateFlow и StateFlow?

*   **MutableStateFlow** — это изменяемый поток данных. Он позволяет изменять своё значение. Используется внутри ViewModel для обновления состояния.
*   **StateFlow** — это неизменяемый поток данных. Предоставляет доступ только для чтения к текущему состоянию.
*   **Зачем создавать два поля: `_uiState` и `uiState`?**
    *   Это реализация паттерна "backing property" (резервное свойство). Приватное поле `_uiState` (тип `MutableStateFlow`) используется только внутри ViewModel для изменения состояния. Публичное поле `uiState` (тип `StateFlow`) предоставляется UI для наблюдения. Это гарантирует, что UI может только читать состояние, но не может его напрямую изменять, что предотвращает неконтролируемые мутации данных и улучшает инкапсуляцию.

### 3. Почему в ViewModel нельзя хранить Context или View?

*   **Что может произойти, если сохранить ссылку на Activity в ViewModel?**
    *   ViewModel живёт дольше, чем Activity. Если сохранить ссылку на Activity (или любой другой Context) в ViewModel, то это создаст "утечку памяти" (memory leak). Уничтоженная Activity не сможет быть собрана сборщиком мусора (GC), так как на неё всё ещё будет ссылаться ViewModel, что приведёт к неэффективному использованию памяти и возможным сбоям приложения.

### 4. Чем StateFlow отличается от remember и rememberSaveable?

*   **StateFlow** — используется на уровне ViewModel и всей архитектуры приложения. Позволяет управлять состоянием в едином источнике правды (Single Source of Truth) и подходит для хранения сложных данных, которые должны жить вне жизненного цикла Composables.
*   **remember** — хранит состояние только внутри конкретной Composable-функции и теряет его при её пересоздании (например, при повороте экрана).
*   **rememberSaveable** — похож на `remember`, но сохраняет состояние при изменении конфигурации (повороте) через `Bundle`. Однако он не подходит для сложной логики и не обеспечивает единый источник правды.
*   **Когда лучше использовать каждый подход?**
    *   Для глобального состояния приложения, бизнес-логики и данных, которые должны сохраняться при повороте, используйте **StateFlow в ViewModel**.
    *   Для простого, локального состояния (например, анимация, выбранная вкладка), которое не нужно сохранять при повороте, используйте `remember`.

### 5. Что такое data class в Kotlin и зачем он используется для UI State?

*   `data class` — это специальный класс в Kotlin, предназначенный для хранения данных.
*   **Какие методы автоматически генерируются?**
    *   `equals()`, `hashCode()`, `toString()`, `copy()`, `componentN()`.
*   **Зачем нужна функция copy()?**
    *   Функция `copy()` используется для создания копии объекта с изменёнными полями. В архитектуре с неизменяемым (immutable) состоянием (как в данном проекте), вместо изменения существующего объекта мы создаём новый объект с обновлёнными значениями. Это делает код более предсказуемым и удобным для отслеживания изменений.

### 6. Как работает collectAsState() в Compose?

*   `collectAsState()` — это функция-расширение для `StateFlow`. Она используется в Composable-функциях для подписки на поток данных.
*   **Почему UI автоматически обновляется при изменении StateFlow?**
    *   `collectAsState()` собирает значения из `StateFlow` и преобразует их в `State<T>`, который является наблюдаемым (observable) для Compose. Когда значение в `StateFlow` изменяется, `collectAsState()` запускает перекомпозицию (recomposition) всех Composable-функций, которые читают это состояние, автоматически обновляя UI.

### 7. Зачем разделять код на пакеты data и ui?

*   Разделение на пакеты (пакеты `data` и `ui` в данном проекте) — это реализация принципов чистой архитектуры.
*   **Какие принципы архитектуры это поддерживает?**
    *   **Разделение ответственности (Separation of Concerns)**: Пакет `data` отвечает за модели данных (структуры, константы), а пакет `ui` — за логику отображения и управления интерфейсом (ViewModel, Composable-функции).
    *   **Инкапсуляция**: Код, отвечающий за разные аспекты приложения, изолирован, что упрощает его поиск, поддержку и тестирование.

### 8. Что происходит с ViewModel, когда приложение полностью закрывается (process death)?

*   **Сохраняются ли данные в этом случае?**
    *   Нет, ViewModel не сохраняет данные при полном завершении процесса приложения (когда система убивает приложение для освобождения ресурсов или когда пользователь закрывает его из меню последних приложений).
*   **Какие альтернативы есть для долговременного хранения?**
    *   Для долговременного хранения данных используются другие механизмы:
        *   **DataStore**: Для хранения небольших наборов данных (например, настроек пользователя).
        *   **Room Database**: Для хранения структурированных данных в локальной базе SQLite.
        *   **Saved State Module для ViewModel**: Для сохранения и восстановления критического состояния при "process death" можно использовать `SavedStateHandle`.
